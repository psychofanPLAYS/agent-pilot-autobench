# World-Class Results Page — Design Spec

**Date:** 2026-06-29
**Status:** Draft for review
**Goal:** Turn pilotBENCHY's flat, table-only results output into a world-class,
diagram-rich results experience — built on a principled metrics model that is
ready to sync online for multi-user, multi-model testing.

---

## 1. Problem

Today the benchmark app produces two HTML pages, both **tables only, zero charts**:

- **Global dashboard** — `runs/results.html`, generated by
  `reports.py::_leaderboard_html`. Hero verdict + model-comparison matrix +
  all-runs table + winning settings.
- **Per-run report** — `<receipt>/report.html`, generated by
  `run_report.py::_html`. Verdict + attempts table + settings + metric coverage.

The data underneath is rich, but the presentation buries the story. There is also
no single headline quality number, no visual "at a glance" comparison, no trend
view across runs, and no schema designed for eventual online sync.

The user's own recorded validity concerns apply here: scoring formulas diverge,
there is no N-repeat with variance, and partial runs can overclaim. A world-class
metric must standardize the formula and surface coverage + variance.

## 2. What the industry leaders do (research synthesis)

All strong players organize model data around **three axes** and fuse them in one
signature scatter:

- **Artificial Analysis** — *Intelligence Index* (one composite 0–100 from ~8
  evals); Output Speed (tok/s), TTFT, Total-time-for-100-tokens (synthetic,
  standardized), **single vs 10-parallel** load, **median (P50) over rolling
  windows** (72h standard, 14d for 100k-context). Signature charts:
  **Quality-vs-Price** and **Quality-vs-Speed frontier** scatter.
- **OpenRouter** — usage/market-share **trend lines over time** (weekly), task
  leaderboards by spend share, per-model throughput/latency/uptime%/context.
- **Bench360** (local-inference academic taxonomy) — adds the local-only
  dimensions: TTFT, inter-token latency (ITL), TPS, requests/sec, **peak VRAM**,
  **power (W), tokens/Joule**, GPU/CPU utilization; explicit **single-user vs
  multi-user/batch** split.

**Three takeaways:**
1. A single composite headline number makes the page legible.
2. The quality-vs-speed frontier scatter (bubble = size) is the defining diagram.
3. pilotBENCHY owns the hardware, so it can report **real VRAM / energy
   efficiency** that cloud dashboards cannot — our differentiator, not a copy.

## 3. The metrics model

Every metric is recorded as a typed record: `{ value, unit, method, load_scenario,
sample_size, status }`. Metrics fall on three axes.

### 3.1 Quality axis (hardware-independent → poolable across users)

**Agent Index (0–100)** — the headline. Composition follows the **mid-2026 gold
standard** (Artificial Analysis Intelligence Index v4.1): a **category-weighted
average**, not a flat mean, with **transparency-forward** reporting (show the
components, not just the composite) — the cutting-edge best practice that answers
the standing critique that composites hide thin evidence.

**Categories and default weights** (agent-worker analogue of AA's Agents-heavy
split). Each signal is an accuracy already in `[0,1]`, so min-max normalization is
the identity here — no hidden "normalization fudging," which is itself the
transparent choice:

| Category | Weight | Signals |
|---|---|---|
| Agentic execution | 38% | benchmark-suite `agentic` score, `librarian-triage`, `librarian-write-entry` |
| Retrieval & synthesis | 30% | `librarian-rerank`, `librarian-compress`, `librarian-dedupe` |
| Reasoning & correctness | 32% | `librarian-contradiction`, benchmark-suite `general` score |

`Agent Index = 100 × Σ_category ( weight × mean(category signals present) )`,
weights renormalized over **categories that have ≥1 measured signal**.

Rules:
- **Gate is a must-pass prerequisite, not a weighted term.** `librarian-gate` acts
  as a multiplier: pass → ×1; fail → the model is flagged `gate-failed` and the
  index is capped (≤ 40) regardless of other categories.
- **Single canonical formula** in one function. No competing score formulas.
- **Transparency block** travels with the index: every category sub-score, every
  raw component, the **coverage %** (measured ÷ intended signals), and the
  **confidence interval** from N-repeat. A 1-signal index and a full-coverage index
  are never shown as equivalent.
- Index is hardware-independent → across synced machines it is **poolable**
  (community mean per model, weighted by coverage).

### 3.1a Standardized variance — applies to ALL metrics

Repeat-with-variance is a **global standard**, not quality-only. Every metric
(quality *and* speed: tok/s, TTFT, etc.) is sampled **K=3 by default** and stored
as **median + IQR** (P25/P75). Single-shot values are marked `unreplicated`. This
fixes the recorded "no N-repeat" validity gap uniformly and makes both the index
CI and the speed bars honest about spread. K is configurable (lower for fast
smoke, higher for publishable runs).

### 3.2 Speed axis (hardware-dependent → bucket by GPU when synced)

- Output speed (generation tok/s)
- Prefill speed (prompt tok/s)
- TTFT cold (ms), TTFT warm (ms), warmup penalty (ms)
- Inter-token latency (ms) — derived
- **Total time for 100 output tokens (s)** — synthetic, standardized (AA-style)
- Serving tok/s under **single** and **parallel (N=10)** load

### 3.3 Efficiency axis (the differentiator)

- Peak VRAM during run (GB) — captured live, persisted per run
- **tok/s per GB VRAM**
- **Quality-per-GB** (Agent Index ÷ peak VRAM)
- **Tokens/Joule** — via `nvidia-ml-py` power sampling (already a dependency);
  omitted gracefully when power is unavailable
- Context retention = tok/s at max successful context ÷ baseline tok/s

### 3.4 Identity / provenance (makes sync possible)

Each record carries stable identifiers so results from many machines/users can be
merged and bucketed:

- `model_fingerprint` — name + params + quant + file size (+ sha when cheap)
- `machine_fingerprint` — GPU name, VRAM total, CPU, OS
- `run_id`, `timestamp` (ISO-8601 UTC)
- `tool_versions` — pilotBENCHY version, llama.cpp build, key flags
- `schema_version`

### 3.5 Data sources (mostly already collected)

| Metric | Source today |
|---|---|
| gen/prompt tok/s | `result.generation/prompt_tokens_per_second` (llama-bench) |
| TTFT cold/warm, serving tok/s | `result.serving_*` |
| pack accuracies | receipt `results.json` / `librarian-suite-summary.json` |
| suite scores | `benchmark_suite_general/agentic_score` |
| context scaling, retention | `context-profile.json` |
| quality drift | `perplexity-profile.json` |
| VRAM | telemetry `gpu_used_mb` / `gpu_total_mb` (needs per-run peak persistence) |
| power | `nvidia-ml-py` power draw (new sampling) |
| run history (trends) | `run_history.py` |

**New work:** Agent Index function; **K=3 repeat-sampling harness with median+IQR
standardized across all metrics**; per-run peak-VRAM + power capture; the
normalized `metrics.json` writer; total-time-100 derivation.

## 4. The diagrams

Charting via a **vendored Chart.js** (`src/gguf_limit_bench/assets/chart.umd.min.js`,
MIT, inlined into the HTML — fully offline, opens from `file://`). All chart logic
lives in a new isolated module `charts.py`.

### 4.1 Global dashboard (`results.html`)

1. **KPI strip** — champion, Agent Index, gen speed, efficiency (tok/s/GB), models·runs.
2. **Quality-vs-Speed frontier scatter** (hero) — Agent Index (y) vs gen tok/s (x),
   bubble = VRAM/params, color = family.
3. **Agent Index leaderboard** — ranked horizontal bars, red→green by score.
4. **Per-pack capability radar** — one polygon per model across the packs.
5. **Speed breakdown** — grouped bars (gen / prompt / serving) and TTFT cold/warm.
6. **Efficiency frontier** — quality-per-GB scatter/bars.
7. **Trend map** — Agent Index & speed across runs over time (from `run_history`).
8. Existing model-comparison matrix + all-runs tables kept as drill-down, color-keyed.

Graceful degradation: charts with no data are omitted; "no agent scores yet"
shows a speed-only hero.

### 4.2 Per-run report (`report.html`)

1. **Context-scaling trend line** — context size → gen tok/s (+ TTFT 2nd axis) and
   quality retention.
2. **Attempts progression** — score per optimizer attempt.
3. **Outcome doughnut + per-pack accuracy bars** — only when the receipt has
   `results.json`; omitted otherwise.
4. Existing attempts table, settings, metric coverage retained.

## 5. Architecture

- `metrics.py` (new) — pure: metric records, Agent Index, coverage, N-repeat
  aggregation, `metrics.json` build/write. No I/O beyond the writer. Fully TDD'd.
- `telemetry.py` (extend) — per-run peak VRAM + optional power sampling helper.
- `charts.py` (new) — `chartjs_runtime()` (inlined lib + dark theme) and pure
  config builders (`quality_vs_speed_scatter`, `agent_index_bar`, `pack_radar`,
  `speed_bars`, `efficiency_frontier`, `index_trend`, `context_scaling_line`,
  `attempts_progression`, `outcome_doughnut`, `pack_accuracy_bars`) +
  `render_chart(id, config, height)`. Builders are the testable seam.
- `reports.py` / `run_report.py` (extend) — consume `metrics.py` + `charts.py`;
  inject KPI strip + chart sections above existing tables.
- `assets/chart.umd.min.js` — vendored library (committed; ships in the wheel via
  hatchling's default package-file inclusion).

Unit boundaries: a builder takes a payload dict, returns a Chart.js config dict —
understandable and testable without touching HTML or the vendored lib.

## 6. Testing

- `tests/test_metrics.py` — Agent Index formula (weights, gate must-pass, coverage,
  renormalization, N-repeat median/IQR, empty/partial inputs).
- `tests/test_charts.py` — each builder's config shape, labels, color mapping,
  empty-data handling.
- Extend `test_reports.py` + new `test_run_report.py` — HTML contains the chart
  canvases + inlined runtime; still renders with zero/partial data.
- TDD: write the failing test first for every pure function in `metrics.py` and
  `charts.py`.

## 7. Phasing (decomposition)

- **Phase 1 — Metrics foundation** (this spec's core): `metrics.py`, Agent Index,
  per-run peak-VRAM/power capture, `metrics.json`. No UI risk; pure + tested.
- **Phase 2 — World-class pages**: `charts.py` + vendored Chart.js + upgraded
  `results.html` & `report.html` with all diagrams + trend maps.
- **Phase 3 — Online sync** (future, designed-not-built): a sync client POSTs
  `metrics.json` records to a results service; community leaderboard buckets speed
  by GPU and pools hardware-independent Agent Index. The schema in §3 is the
  contract that makes Phase 3 additive.

## 8. Scope guards (YAGNI)

- No changes to the cockpit `INDEX_HTML` beyond linking to the upgraded pages.
- No sync server built in Phases 1–2; only the sync-ready record.
- No new benchmark *workloads*; we model and visualize what runs already produce
  (plus VRAM/power capture).

## 9. Resolved decisions

1. **Agent Index weights** — category-weighted (AA v4.1-style), transparency-forward
   with components + coverage + CI; gate must-pass (see §3.1).
2. **N-repeat** — **K=3 median + IQR, standardized across ALL metrics** (§3.1a).
3. **Power/energy** — **in Phase 1** with graceful fallback when power is
   unreadable (user: "do it, in parallel, I don't care how — as long as it's done").
4. **Chart.js vendor** — confirmed: commit MIT JS into the package, inline it.

## 10. Design vision — near future (captured, not built in this spec)

The user flagged two larger directions that this metrics foundation must enable.
They are out of scope for Phases 1–2 implementation but the schema and modules are
designed so they slot in additively.

### 10.1 Adaptive auto-planner ("the program that chooses for itself")

A planner that, given **run history** (`run_history.py`) and **available Hugging
Face metadata** for the selected model(s), decides *for itself*:
- which tests/packs to run (skip what's already well-established for this model),
- how long / how many repeats (K) to spend per signal for a target confidence,
- and shows a live **ETA** before and during the run.

Hooks this spec provides: standardized per-signal records with coverage + CI mean
"confidence reached" is queryable; the metrics schema + history are the planner's
inputs. The Agent Index coverage/CI is the natural stopping criterion.

### 10.2 In-flight testing page (preflight → live run experience)

Today launching a run leaves the user on a static page. Vision: the **pre-flight
page transforms into a beautiful in-flight testing view** that shows what's
happening behind the scenes in real time:
- streamed **questions**, model **thinking** (when enabled for the run), and
  **answers**,
- live **insights**, running **score**, and **history** context,
- progress + ETA from the auto-planner (§10.1).

This is a separate page/redesign (the cockpit run flow), not `results.html`. The
event stream (`events.jsonl`, the existing WS feed) and the per-signal metric
records are the data source; the Agent Index can update live as signals land.
Tracked as the next major UI effort after the results pages ship.
